# Workflow Outputs — Results, Deliverables, and Downstream Triggers

## What Comes Out of a Workflow?

Every workflow produces an output — the tangible result of all transformation steps. Outputs can be data records, sent messages, generated documents, status changes, or triggers for downstream workflows. Well-designed outputs are structured, auditable, and useful to both humans and systems.

## Output Categories

### Data Outputs
Records created or modified in databases:
- New appointment record with confirmation number
- Updated customer status (active → churned)
- Created invoice with line items
- Generated KB document ingested into vector store

### Communication Outputs
Messages sent to humans or systems:
- SMS confirmation: "Your appointment is booked for March 20 at 9 AM"
- Email with invoice PDF attached
- Slack notification to technician with job details
- Webhook callback to external system

### Document Outputs
Files generated by the workflow:
- PDF invoice or receipt
- CSV export of processed data
- Generated image or video
- Signed contract or agreement

### Status Outputs
State changes that downstream systems observe:
- Job status changed from `running` to `completed`
- Ticket priority escalated from `medium` to `urgent`
- Payment status moved from `pending` to `captured`
- Feature flag toggled from `false` to `true`

## Output Structure

```typescript
type WorkflowOutput = {
  // Execution identity
  executionId: string;
  workflowId: string;
  tenantId: string;

  // Result
  status: "completed" | "failed" | "partial" | "cancelled";
  data: Record<string, unknown>;

  // Metrics
  startedAt: Date;
  completedAt: Date;
  duration_ms: number;
  stepsExecuted: number;
  stepsFailed: number;

  // Audit
  auditTrail: Array<{
    step: string;
    action: string;
    timestamp: Date;
    success: boolean;
  }>;

  // Downstream
  triggers?: Array<{
    workflowId: string;
    data: Record<string, unknown>;
  }>;
};
```

## Output as Input — Workflow Chaining

The most powerful pattern: one workflow's output triggers another:

```
Workflow A: Process Voice Call
  Output: { appointmentId: "apt-123", customerId: "c-456" }
    ↓ triggers
Workflow B: Send Confirmations
  Output: { smsSent: true, slackNotified: true }
    ↓ triggers
Workflow C: Schedule Follow-Up
  Output: { reminderScheduled: "2025-03-19T08:00:00Z" }
```

Atlas UX chains workflows through the job queue — completing one job can enqueue the next.

## Partial Outputs

When a workflow partially succeeds, report what completed and what didn't:

```json
{
  "status": "partial",
  "data": {
    "appointmentCreated": true,
    "appointmentId": "apt-123",
    "smsConfirmation": true,
    "slackNotification": false,
    "slackError": "Webhook URL not configured"
  },
  "stepsExecuted": 7,
  "stepsFailed": 1
}
```

This lets downstream systems decide how to handle the partial result — retry the failed step, escalate, or accept the partial output.

## Output Validation

Before finalizing, validate the output meets expectations:

```typescript
function validateOutput(output: WorkflowOutput): boolean {
  // Business rule: appointment workflows must produce an appointmentId
  if (output.workflowId === "book-appointment" && !output.data.appointmentId) {
    throw new Error("Appointment workflow completed without appointmentId");
  }

  // All workflows must have audit trail
  if (!output.auditTrail || output.auditTrail.length === 0) {
    throw new Error("Workflow completed without audit trail");
  }

  return true;
}
```

## Output Storage and Retention

| Output Type | Storage | Retention | Reason |
|-------------|---------|-----------|--------|
| Execution results | Database | 90 days | Debugging, analytics |
| Audit trail | Immutable log | 7 years | Compliance (SOC 2) |
| Generated documents | S3/CDN | Per policy | Business records |
| Sent messages | Audit log | 1 year | Communication records |
| Error details | Structured logs | 30 days | Incident investigation |
